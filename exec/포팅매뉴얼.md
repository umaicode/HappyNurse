### Frontend (Next.js 빌드 인자)

Frontend는 환경변수가 **빌드 시점에 정적 자산에 박히는** 구조이므로 `env_file`이 아닌 **Jenkinsfile의 docker `--build-arg`**로 주입됩니다.

```groovy
// Jenkinsfile (자동 처리)
def basePath = env.DEPLOY_ENV == 'dev' ? '/dev' : ''
def apiUrl   = env.DEPLOY_ENV == 'dev' ? 'https://k14e101.p.ssafy.io/dev/api' : 'https://k14e101.p.ssafy.io/api'
def aiUrl    = env.DEPLOY_ENV == 'dev' ? 'https://k14e101.p.ssafy.io/dev/ai'  : 'https://k14e101.p.ssafy.io/ai'

def buildArgs = [
    "--build-arg NEXT_PUBLIC_BASE_PATH=${basePath}",
    "--build-arg NEXT_PUBLIC_API_BASE_URL=${apiUrl}",
    "--build-arg NEXT_PUBLIC_AI_BASE_URL=${aiUrl}",
    "--build-arg NEXT_PUBLIC_APP_ENV=${env.DEPLOY_ENV}",
].join(' ')
```

| 변수 | dev | prod | 비고 |
|---|---|---|---|
| `NEXT_PUBLIC_BASE_PATH` | `/dev` | (빈 값) | Next.js basePath |
| `NEXT_PUBLIC_API_BASE_URL` | `.../dev/api` | `.../api` | Backend URL |
| `NEXT_PUBLIC_AI_BASE_URL` | `.../dev/ai` | `.../ai` | AI URL |
| `NEXT_PUBLIC_APP_ENV` | `dev` | `prod` | UI 표기 분기 |

## 1-3. 배포 시 특이사항

### (1) 단일 도메인 path 기반 라우팅

```
https://k14e101.p.ssafy.io/                  → Prod Frontend (Next.js)
https://k14e101.p.ssafy.io/api/              → Prod Backend (Spring Boot)
https://k14e101.p.ssafy.io/ai/               → Prod AI (FastAPI)
https://k14e101.p.ssafy.io/dev/              → Dev Frontend
https://k14e101.p.ssafy.io/dev/api/          → Dev Backend
https://k14e101.p.ssafy.io/dev/ai/           → Dev AI
https://k14e101.p.ssafy.io/jenkins/          → Jenkins UI
https://k14e101.p.ssafy.io/grafana/          → Grafana UI
https://k14e101.p.ssafy.io/.well-known/assetlinks.json  → Android App Links 검증
```

### (2) Blue/Green 무중단 배포

각 서비스(backend / ai / frontend)는 dev/prod별로 **blue / green 두 컨테이너 슬롯**을 가집니다. `deploy.sh` 스크립트가:

1. 현재 활성 색(예: blue) 자동 감지
2. 반대 색(green) 컨테이너를 새 이미지로 기동
3. Nginx 컨테이너의 `nc`로 외부 헬스체크 (최대 120초 대기)
4. 헬스체크 통과 시 Nginx upstream 파일을 새 색으로 교체 + `nginx -s reload`
5. 옛 색 컨테이너 10초 그레이스 후 제거
6. **헬스체크 실패 시 새 컨테이너 폐기, 옛 색 그대로 유지 → 자동 롤백**

### (3) Nginx upstream 동적 resolve

각 서비스 컨테이너가 잦은 교체로 일시 부재할 수 있으므로, **정적 upstream 블록이 아닌 `resolver` + `map 변수` 패턴**을 사용해 요청 시점에 DNS 해석합니다.

```nginx
# nginx.conf
resolver 127.0.0.11 valid=10s ipv6=off;

# conf.d/upstream-backend-dev.conf
map "" $backend_dev_target {
    default "backend-dev-blue:8080";   # deploy.sh가 동적 교체
}
```

→ 한 서비스가 일시 부재해도 다른 서비스의 reload가 차단되지 않음.

### (4) URL prefix 책임은 인프라 레이어

- Backend / AI 코드는 `/api`, `/ai` prefix 없이 **root 기준으로 작성**
- Nginx가 `proxy_pass http://...$1$is_args$args;` 패턴으로 prefix 제거 후 전달
- → 같은 이미지로 dev/prod 동일하게 배포 가능

### (5) Frontend는 빌드타임 분리 필수

Next.js의 `basePath`는 정적 자산 경로에 빌드타임에 박히므로 **환경별로 다른 이미지** 빌드:

- dev: `NEXT_PUBLIC_BASE_PATH=/dev`로 빌드
- prod: 빈 값으로 빌드

### (6) Webhook 트리거 (GitLab → Jenkins)

| Jenkins Job | Source Branch (regex) | Target Branch (regex) | 이벤트 |
|---|---|---|---|
| dev-deploy | `.*` | `develop` | Push + Merge Request |
| prod-deploy | `.*` | `master` | Merge Request only |

> Source Branch Regex를 빈칸 두면 매칭 실패. **반드시 `.*` 명시**.

### (7) Detect Changes (변경 서비스만 선택 빌드)

Jenkinsfile이 `GIT_PREVIOUS_SUCCESSFUL_COMMIT`과 현재 commit의 diff를 비교해, 다음 디렉토리 변경 시에만 해당 서비스를 빌드:

- `backend/` → Backend 빌드
- `ai/` → AI 빌드
- `frontend/web/` → Frontend 빌드
- `Jenkinsfile` 또는 `infra/` → 전체 재빌드

### (8) Firebase 키 마운트 권한

```bash
chmod 700 /home/deploy/infra/secrets/                                # 디렉토리 = 게이트
chmod 644 /home/deploy/infra/secrets/firebase/service-account.json  # 컨테이너 spring 유저(UID 999)도 읽기 가능
```

호스트의 deploy 유저(UID 1000) ≠ 컨테이너의 spring 유저(UID 999)이므로 600 권한은 컨테이너에서 못 읽음. **디렉토리 게이트 + 파일 644 다층 방어**.

### (9) Database Timezone

모든 DB / 컨테이너 timezone은 `Asia/Seoul`로 통일:

```sql
ALTER DATABASE happynurse_dev SET timezone TO 'Asia/Seoul';
ALTER DATABASE happynurse_prod SET timezone TO 'Asia/Seoul';
```

각 컨테이너 env 파일에 `TZ=Asia/Seoul`, PostgreSQL은 `PGTZ=Asia/Seoul` 추가.

### (10) ShedLock 테이블 수동 생성 필요

ShedLock 라이브러리 테이블은 JPA Entity가 아니므로 `ddl-auto`로 자동 생성되지 않음. **DB 초기화 시 수동 생성 필수**:

```sql
CREATE TABLE IF NOT EXISTS shedlock (
    name VARCHAR(64) NOT NULL,
    lock_until TIMESTAMP NOT NULL,
    locked_at TIMESTAMP NOT NULL,
    locked_by VARCHAR(255) NOT NULL,
    PRIMARY KEY (name)
);
```

### (11) Android App Links 검증 파일

`/home/deploy/infra/nginx/well-known/assetlinks.json`에 release / debug keystore 양쪽의 SHA-256 fingerprint 등록 필요. Nginx 설정으로 `/.well-known/assetlinks.json` 200 OK 응답 보장.

### (12) HTTPS 인증서

Let's Encrypt 발급. 90일마다 갱신 필요. cron으로 매주 일요일 새벽 3시 자동 갱신:

```cron
0 3 * * 0 docker run --rm -v /home/deploy/infra/nginx/certs:/etc/letsencrypt -v /home/deploy/infra/nginx/certbot-webroot:/var/www/certbot certbot/certbot renew --quiet && docker exec nginx nginx -s reload
```

# 2. 외부 서비스 정보

본 프로젝트에서 활용한 외부 서비스 목록과 각 서비스 가입·설정에 필요한 정보입니다.

### 2-1. Firebase Cloud Messaging (FCM)

- **용도**: Android 앱 푸시 알림 (IV 교체 알림, 환자 인증 알림 등)
- **가입 / 콘솔**: https://console.firebase.google.com/
- **필요한 작업**:
  1. Firebase 프로젝트 생성
  2. **Android 앱 등록** (package name: `com.happynurse`, SHA-1 fingerprint)
  3. `google-services.json` 다운로드 → `frontend/mobile/app/` 에 배치
  4. **Service Account Key 생성** (Project Settings → Service Accounts → Generate new private key)
  5. 다운받은 JSON 파일을 `/home/deploy/infra/secrets/firebase/service-account.json`에 배치
  6. 권한 설정: 디렉토리 700, 파일 644
- **사용 환경변수**:
  - Backend: `FIREBASE_CREDENTIALS_PATH=file:/app/firebase/service-account.json`
- **요금**: 무료 (Spark 플랜)

### 2-2. Naver CLOVA Speech Recognition (STT)

- **용도**: 간호사 음성 메모를 텍스트로 변환 (간호기록 자동 생성)
- **가입 / 콘솔**: https://www.ncloud.com/product/aiService/clovaSpeechRecognition
- **필요한 작업**:
  1. 네이버 클라우드 플랫폼 가입
  2. Application 등록 → CLOVA Speech (Long Sentence) 활성화
  3. **Secret Key** 및 **Invoke URL** 발급
- **사용 환경변수** (AI):
  - `CLOVA_SECRET_KEY=<Secret>`
  - `CLOVA_INVOKE_URL=<Invoke URL>`
- **요금**: 종량제 (월 60분 무료 → 이후 분당 과금)

# 3. DB 덤프 파일

### 3-1. 덤프 파일 위치

본 프로젝트는 PostgreSQL 16 기반. 최신 덤프 파일은 다음 경로에 포함되어 있습니다:

```
exec/
└── db_dump/
    ├── happynurse_prod_<날짜>.sql      # prod 환경 전체 덤프
    └── happynurse_dev_<날짜>.sql       # dev 환경 전체 덤프 (참고용)
```

### 3-2. 덤프 생성 명령 (운영 환경에서 추출)

```bash
# Prod DB 덤프
docker exec postgres-prod pg_dump -U happynurse_app -d happynurse_prod \
  --clean --if-exists --no-owner --no-privileges \
  > happynurse_prod_$(date +%Y%m%d).sql

# Dev DB 덤프
docker exec postgres-dev pg_dump -U happynurse_app -d happynurse_dev \
  --clean --if-exists --no-owner --no-privileges \
  > happynurse_dev_$(date +%Y%m%d).sql
```

### 3-3. 덤프 복원 명령

```bash
# 1) DB 초기화 (선택)
docker exec -i postgres-prod psql -U happynurse_app -d happynurse_prod <<'EOF'
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
GRANT ALL ON SCHEMA public TO happynurse_app;
GRANT ALL ON SCHEMA public TO public;
EOF

# 2) 덤프 복원
docker exec -i postgres-prod psql -U happynurse_app -d happynurse_prod < happynurse_prod_<날짜>.sql

# 3) ShedLock 테이블 생성 (JPA 외 라이브러리 테이블)
docker exec -i postgres-prod psql -U happynurse_app -d happynurse_prod <<'EOF'
CREATE TABLE IF NOT EXISTS shedlock (
    name VARCHAR(64) NOT NULL,
    lock_until TIMESTAMP NOT NULL,
    locked_at TIMESTAMP NOT NULL,
    locked_by VARCHAR(255) NOT NULL,
    PRIMARY KEY (name)
);
EOF

# 4) timezone 설정
docker exec -i postgres-prod psql -U happynurse_app -d happynurse_prod <<'EOF'
ALTER DATABASE happynurse_prod SET timezone TO 'Asia/Seoul';
EOF
```
